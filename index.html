Intelligent Cloud Operations Agent

LangChainPythonTerraformFastAPI

Problem Statement

Cloud operations teams rely on manual queries for routine diagnostics. This project aimed to create an AI agent to automate these tasks using natural language, providing instant Infrastructure-as-Code (IaC) awareness.

Architecture

Technical Decisions & Outcomes

  • Architectural Choice: A self-hosted Hugging Face model on a GCE VM was chosen over proprietary APIs to ensure data privacy and eliminate API costs, accepting the trade-off of increased management overhead.
  • Technical Challenge: Securely parsing the Terraform state file was solved by creating a dedicated Service Account with only the `storage.objects.get` permission on the specific `.tfstate` object, adhering to the principle of least privilege.
  • Quantifiable Outcome: Automated 5 core operational checks, reducing the time for routine diagnostics from 5-10 minutes manually to **under 15 seconds** via natural language.

Real-Time Serverless Data Pipeline

Pub/SubCloud FunctionsBigQueryPython

Problem Statement

IoT devices generate high-volume data streams that require immediate ingestion, validation, and analysis. This project's goal was to build a highly scalable, low-latency, and cost-effective serverless pipeline to handle this workflow.

Architecture

Technical Decisions & Outcomes

  • Architectural Choice: Cloud Functions Gen 2 was selected over Gen 1 for its superior event-driven capabilities, including built-in Pub/Sub triggers and longer processing timeouts, making it ideal for potentially complex validation logic without needing a containerized environment like Cloud Run.
  • Technical Challenge: The core challenge was designing stateless, idempotent validation logic. The function was built to handle out-of-order or duplicate messages gracefully, ensuring that data anomalies (e.g., negative sensor readings) were flagged and routed to a secondary Pub/Sub topic for alerting without halting the entire pipeline.
  • Quantifiable Outcome: The pipeline was architected to handle **1,000+ messages per second** with an average end-to-end latency (ingestion to BigQuery storage) of **under 2 seconds**, all while operating within GCP's zero-cost free tier for the expected data volume.

Zero-Touch CI/CD Pipeline

GitHub ActionsTerraformDockerCloud Run

Problem Statement

Manual deployments are slow, error-prone, and unscalable. This project aimed to create a fully automated CI/CD pipeline to achieve zero-touch build, test, and deployment of a containerized application to a serverless platform.

Architecture

Technical Decisions & Outcomes

  • Architectural Choice: Google Cloud Run was the ideal deployment target over GKE due to its fully managed, serverless nature. This eliminated Kubernetes management overhead and provided "scale-to-zero" capability, making it highly cost-effective for a web application with variable traffic.
  • Technical Challenge: Configuring secure, least-privilege IAM roles for the GitHub Actions pipeline was paramount. This was solved using Workload Identity Federation, which allows GitHub Actions to securely impersonate a GCP service account using short-lived credentials, completely avoiding static, long-lived keys.
  • Quantifiable Outcome: The pipeline fully automated the deployment lifecycle, reducing the process from manual steps taking 30+ minutes to an average, zero-touch execution time of **4 minutes and 15 seconds** from `git push` to live deployment.

Production-Grade Service Mesh

GKEIstioTerraformmTLSObservability

Problem Statement

In a microservices architecture, ensuring secure communication, fault tolerance, and deep observability is complex. This project demonstrates a production-grade service mesh on GKE to solve these challenges using advanced cloud-native patterns.

Architecture

Technical Decisions & Outcomes

  • Architectural Choice: A canary deployment strategy was chosen over blue/green to demonstrate a more sophisticated, risk-averse release process. This allows for routing a small subset of live traffic (e.g., 10%) to a new version, enabling production testing with minimal user impact before a full rollout.
  • Technical Challenge: The most complex task was writing an Istio `AuthorizationPolicy` to enforce a zero-trust security posture. A policy was created to deny all traffic by default and explicitly only allow service-to-service communication on specific paths and HTTP methods (e.g., allow `GET` from `frontend` to `api-service` but deny `POST`).
  • Quantifiable Outcome: The entire GKE cluster, application, and Istio mesh setup was automated with Terraform, reducing deployment time from **over 4 hours** of manual steps to a repeatable, scripted deployment of **under 9 minutes**.
