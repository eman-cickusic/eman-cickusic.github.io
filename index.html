Intelligent Cloud Operations Agent

LangChainPythonTerraformFastAPI

Problem Statement

Cloud operations teams rely on manual queries for routine diagnostics. This project aimed to create an AI agent to automate these tasks using natural language, providing instant Infrastructure-as-Code (IaC) awareness.

Architecture

Technical Decisions & Outcomes

  • Architectural Choice: A self-hosted Hugging Face model on a GCE VM was chosen over proprietary APIs to ensure data privacy and eliminate API costs, accepting the trade-off of increased management overhead.
  • Quantifiable Outcome: Automated 5 core operational checks, reducing the time for routine diagnostics from 5-10 minutes manually to under 15 seconds via natural language.

Security & Cost Optimization

  • Security Posture: Access to the Terraform state file was secured by creating a dedicated GCP Service Account with only the `storage.objects.get` permission on the specific `.tfstate` object, adhering to the principle of least privilege.
  • Cost Optimization: Using a self-hosted open-source LLM on a preemptible GCE instance significantly reduced operational costs compared to using a proprietary, per-token API like GPT-4, making the tool effectively free for internal use.

Testing Strategy

  • Unit Testing: The FastAPI backend logic, including the parsing of natural language queries, was validated using `pytest` to ensure predictable responses.
  • Integration Testing: A dedicated test script was created to verify the agent's connection and functional interaction with live GCP APIs (Logging, Firestore) in a non-production environment.

Real-Time Serverless Data Pipeline

Pub/SubCloud FunctionsBigQueryPython

Problem Statement

IoT devices generate high-volume data streams that require immediate ingestion, validation, and analysis. This project's goal was to build a highly scalable, low-latency, and cost-effective serverless pipeline to handle this workflow.

Architecture

Technical Decisions & Outcomes

  • Architectural Choice: Cloud Functions Gen 2 was selected for its superior event-driven capabilities and longer processing timeouts, making it ideal for complex validation logic without needing a containerized environment like Cloud Run.
  • Quantifiable Outcome: The pipeline was architected to handle 1,000+ messages per second with an average end-to-end latency of under 2 seconds, all while operating within GCP's zero-cost free tier for the expected data volume.

Security & Cost Optimization

  • Security Posture: The Cloud Function was assigned a dedicated, non-default service account with the minimum required roles (`roles/pubsub.publisher`, `roles/bigquery.dataEditor`) to write to the downstream services, preventing privilege escalation.
  • Cost Optimization: The entire architecture is serverless, leveraging a "pay-per-use" model. This means zero cost when idle and automatic scaling during traffic spikes, making it vastly more cost-effective than a provisioned-server solution.

Testing Strategy

  • Unit Testing: The core Python validation logic within the Cloud Function was unit-tested with `pytest`, using mock data to cover edge cases like malformed payloads and invalid sensor readings.
  • End-to-End Testing: A deployment script publishes a test message to the initial Pub/Sub topic and polls the target BigQuery table to confirm the entire pipeline completes successfully within latency targets.

Zero-Touch CI/CD Pipeline

GitHub ActionsTerraformDockerCloud Run

Problem Statement

Manual deployments are slow, error-prone, and unscalable. This project aimed to create a fully automated CI/CD pipeline to achieve zero-touch build, test, and deployment of a containerized application to a serverless platform.

Architecture

Technical Decisions & Outcomes

  • Architectural Choice: Google Cloud Run was the ideal deployment target over GKE due to its fully managed, serverless nature. This eliminated Kubernetes management overhead and provided "scale-to-zero" capability, making it highly cost-effective.
  • Quantifiable Outcome: The pipeline fully automated the deployment lifecycle, reducing the process from manual steps taking 30+ minutes to an average, zero-touch execution time of 4 minutes and 15 seconds from `git push` to live deployment.

Security & Cost Optimization

  • Security Posture: The pipeline was secured using Workload Identity Federation, which allows GitHub Actions to securely impersonate a GCP service account with short-lived credentials, completely avoiding static, long-lived keys.
  • Cost Optimization: Cloud Run's "scale-to-zero" feature means no charges are incurred when the application is not receiving requests. The use of Artifact Registry for storing images is also more cost-effective than third-party registries for intra-GCP traffic.

Testing Strategy

  • CI Stage: The pipeline integrates `pytest` for unit tests and a linter (`flake8`) to enforce code quality, failing the build if checks do not pass.
  • Infrastructure Validation: `terraform validate` and `tflint` are used to check syntax and best practices. A `terraform plan` is posted as a pull request comment for mandatory peer review before any infrastructure changes can be merged and applied.

Production-Grade Service Mesh

GKEIstioTerraformmTLSObservability

Problem Statement

In a microservices architecture, ensuring secure communication, fault tolerance, and deep observability is complex. This project demonstrates a production-grade service mesh on GKE to solve these challenges using advanced cloud-native patterns.

Architecture

Technical Decisions & Outcomes

  • Architectural Choice: A canary deployment strategy was chosen over blue/green to demonstrate a more sophisticated, risk-averse release process, enabling production testing with minimal user impact before a full rollout.
  • Quantifiable Outcome: The entire GKE cluster and Istio mesh setup was automated with Terraform, reducing deployment time from over 4 hours of manual steps to a repeatable, scripted deployment of under 9 minutes.

Security & Cost Optimization

  • Security Posture: A zero-trust network was enforced using an Istio `AuthorizationPolicy` to deny all traffic by default, with explicit rules only allowing required service-to-service communication. All inter-service traffic is automatically encrypted with mTLS by the mesh.
  • Cost Optimization: The GKE cluster was configured with node auto-provisioning and auto-scaling, allowing it to dynamically add or remove nodes based on workload demands, preventing over-provisioning and minimizing costs.

Zero-Trust Policy Example

The following Istio `AuthorizationPolicy` denies all traffic in the namespace by default, then explicitly allows `GET` requests from the `frontend` service account to the `api-service`.

apiVersion: security.istio.io/v1beta1
kind: AuthorizationPolicy
# ... rest of yaml code

Testing Strategy

  • Infrastructure Testing: The Terraform configuration was validated using `tflint` for best practices. The entire GKE cluster and Istio installation were deployed to a staging environment first to validate the setup before any production deployment.
  • Policy Verification: After applying the `AuthorizationPolicy`, tests were run from a non-whitelisted pod to confirm that access to the `api-service` was correctly denied, explicitly verifying the zero-trust configuration.
